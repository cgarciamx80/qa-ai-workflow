## What This Project Does
Simulates a real QA engineering process using AI. Given a user story as input, it runs a 4-stage pipeline:

1. **Planner** — Claude generates a structured test plan (5–8 test cases)
2. **CodeGen** — Claude generates a Playwright `.spec.ts` file from those test cases
3. **Runner** — Playwright executes the tests against `https://demo.playwright.dev/todomvc`
4. **Analyzer** — Claude analyzes the results and generates bug reports in JIRA and Azure DevOps format

## Key Design Decisions & Best Practices Applied

### Structured Outputs (no regex)
Planner and Analyzer use `output_config.format` with a `json_schema` — the SDK
guarantees valid JSON back. No more `.replace(/```json|```/g, "")` hacks.
```typescript
output_config: {
  format: { type: "json_schema", schema: { ...JSONSchema } }
}
```

### Centralized Config (`src/config.ts`)
All constants in one place — model name, URLs, file paths, token limits.
No magic strings scattered through agents.

### Error Handling
- API key check at pipeline startup with a clear message
- Each agent throws a typed error if the API returns no text block
- `spawnSync` error propagation in the runner
- `process.exit(1)` with a readable message on pipeline failure

### Playwright Runner
Uses `spawnSync` instead of the old `execSync` — proper error propagation,
reads results from `output/results.json` written by the JSON reporter.

### .gitignore
Excludes: `node_modules/`, `dist/`, `output/`, `tests/generated.spec.ts`, `.env`

## TypeScript Config
- `module: ESNext`, `moduleResolution: Bundler` — matches how `tsx` resolves modules
- `strict: true`, `skipLibCheck: true`
- No `.js` extensions needed on imports (Bundler mode handles it)

## API Usage Pattern
- Planner + Analyzer: `client.messages.create()` with `output_config` (structured JSON)
- CodeGen: `client.messages.create()` — plain text, strips stray markdown fences
- Model: `claude-opus-4-6` for all agents
